## 1. Tratamiento de Discos en Linux

En Linux, los discos duros y otros medios de almacenamiento de bloques se representan como archivos en el directorio `/dev`.

La nomenclatura depende del tipo de conexiГіn fГӯsica o bus:

- **Discos SATA/SAS (Tradicionales y SSDs):** `/dev/sda` (primer disco), `/dev/sdb` (segundo disco), etc.
- **Discos NVMe (Alto rendimiento PCIe):** `/dev/nvme0n1` (primer disco), `/dev/nvme1n1` (segundo disco).
- **Lectores SD / eMMC:** `/dev/mmcblk0`.

Cuando particionamos un disco, a cada particiГіn se le asigna un nГәmero:

- Las particiones de `/dev/sda` se llaman `/dev/sda1`, `/dev/sda2`, etc.
- Las particiones de `/dev/nvme0n1` se llaman `/dev/nvme0n1p1`, `/dev/nvme0n1p2`, etc.

### MBR vs GPT

- **MBR (Master Boot Record):** Es el estГЎndar antiguo. Limitado a un mГЎximo de 4 particiones primarias y no soporta discos mayores de 2 Terabytes. Se almacena en el primer sector absoluto del disco.
- **GPT (GUID Partition Table):** Es el estГЎndar moderno atado a motherboards UEFI. Permite hasta 128 particiones por defecto en Linux y soporta discos de tamaГұos masivos de Zettabytes. Contiene copias de seguridad de sГӯ misma al final del disco para evitar corrupciГіn.

## 2. Sistemas de Archivos (Filesystems)

Pintar las "rayas de la pista" en un disco para que Linux sepa organizar carpetas requiere formatearlas con un _sistema de archivos_.

- **ext4 (Fourth Extended Filesystem):** Es el estГЎndar tradicional de facto en la familia Debian y la mayorГӯa de Linux. Es rГЎpido, tiene excelente compatibilidad y es _journaling_ (mantiene un registro (journal) de lo que va a escribir antes de hacerlo, para evitar corrupciones si hay un apagГіn).
- **xfs:** El estГЎndar por defecto en la familia RHEL. EstГЎ ultra-optimizado para manejar archivos paralelamente y de tamaГұos gigantescos.
- **btrfs y ZFS:** Sistemas de prГіxima generaciГіn (Copy-on-Write) que gestionan el particionamiento y los archivos simultГЎneamente, permitiendo "snapshots" (fotografГӯas del sistema en el tiempo) nativos sin herramientas extra.

## 4. El Proceso de Arranque (Desde BotГіn de Encendido a Servidor Activo)

ВҝQuГ© pasa desde el segundo que enciendes la mГЎquina de metal hasta que puedes escribir comandos?

```mermaid
graph TD
    A[1. BIOS / UEFI] -->|POST & Hardware Init| B[2. Bootloader / GRUB2]
    B -->|Selects OS| C[3. Linux Kernel]
    C -->|Mounts Initramfs| D[4. Initramfs Temporary Root]
    D -->|Loads Drivers| E[5. Real Root / Mounted]
    E -->|Starts Init Process| F[6. systemd PID 1]
    F -->|Parallel Execution| G[7. Targets & Login]
```

1. **Firmware (BIOS/UEFI):** Inicializa el hardware rudimentario (CPU, memoria, teclado). Comprueba que nada estГЎ en cortocircuito (POST) y busca la particiГіn `/boot/efi` para encontrar el gestor de arranque.
2. **Bootloader (GRUB2):** Lee su configuraciГіn (`/boot/grub/grub.cfg`) que dice dГіnde se encuentra fГӯsicamente el Kernel. Carga el Kernel y un archivo llamado `initramfs` en la memoria RAM, y le cede el control.
3. **El Kernel (`vmlinuz`):** Autodetecta el hardware a bajo nivel. Comienza a gestionar la memoria. Es el cerebro absoluto.
4. **Initramfs (Initial RAM Filesystem):** El Kernel inicialmente no sabe cГіmo leer el disco duro real (`ext4`, volГәmenes LVM, o si estГЎ cifrado con contraseГұa). El `initramfs` es un "mini-Linux" que se carga en la RAM que contiene solo los _drivers_ (mГіdulos) necesarios para saber leer el disco duro central de verdad.
5. **Montaje de RaГӯz:** Una vez superado el `initramfs`, el sistema monta el disco de verdad en la raГӯz (`/`).
6. **Init (`systemd`):** El Kernel ejecuta el primer programa real: `/sbin/init` (que en sistemas modernos es siempre `systemd`). Se le asigna el ID de Proceso 1 (`PID 1`).
7. **Systemd Targets:** `systemd` lee su configuraciГіn e inicia servicios en paralelo para llegar a un "Target" objetivo. Por ejemplo, el `multi-user.target` para servidores. Inicializa la red, SSH, monta bases de datos, y finalmente, muestra el **Prompt de Login**.

<Aside type="caution" title="ВҝQuГ© es una pantalla de Kernel Panic?">
  Si el Kernel no puede encontrar o montar la particiГіn raГӯz (Paso 5) debido a
  que `/etc/fstab` estГЎ mal configurado, el proceso se detiene con un fatal
  *Kernel Panic*. No puede continuar porque no encuentra archivos del disco duro
  que ejecutar. ВЎAprenderemos a diagnosticar esto en mГіdulos futuros!
</Aside>
