## Ejercicio 3.1 — Anatomía de usuarios (UID, passwd/shadow) y ciclo de vida

### Contexto

En LFCS no basta con “crear un usuario”: tienes que **auditar** qué queda escrito en `/etc/passwd`, entender shells, bloquear/desbloquear, y comprobar UIDs/GIDs con precisión.

### Tareas

1) Crea tres cuentas con `useradd` (no `adduser`):

- `audit1` con HOME y shell `/bin/bash`
- `audit2` con HOME y shell `/usr/sbin/nologin`
- `audit3` con HOME y shell `/bin/bash` y un comentario (GECOS) “Cuenta de auditoría”

2) Asigna contraseña a `audit1` y `audit3`. (La cuenta `audit2` no debe poder iniciar sesión interactiva por su shell).

3) Demuestra con evidencia:

- la línea completa de cada usuario en `/etc/passwd`
- el UID y GID principal de cada usuario (con `id`)
- que `/etc/shadow` **no** es legible por un usuario normal (y sí por root)

Guarda comandos y salidas en:

`~/practica3/entrega/31_auditoria_usuarios.txt`

4) Bloquea a `audit3` (sin borrarlo) y demuestra que queda bloqueado; luego desbloquéalo y demuestra que vuelve a estar operativo.

Guarda evidencia (comandos + salida relevante) en:

`~/practica3/entrega/31_lock_unlock.txt`

## Ejercicio 3.2 — Grupos, delegación y sudo mínimo (visudo)

### Contexto

En un servidor real no se comparte root. Delegas poder con grupos y reglas de sudo **mínimas**, auditables y sin romper `/etc/sudoers`.

### Setup — ejecutar una sola vez

```bash
sudo apt update && sudo apt install -y nginx
systemctl status nginx --no-pager
```

### Tareas

1) Crea el usuario `deploy` con HOME y `/bin/bash`. Asigna contraseña.

2) Crea un grupo `webops` y añade `deploy` a ese grupo (sin quitarle otros grupos).

3) Configura sudo para que **solo** el grupo `webops` pueda ejecutar **sin contraseña**:

- `systemctl restart nginx`
- `systemctl status nginx`

Requisitos:

- Debe estar en un archivo drop-in dentro de `/etc/sudoers.d/`
- Debe ser validado con `visudo -c -f`

4) Prueba como `deploy` que:

- `sudo systemctl restart nginx` funciona sin pedir contraseña
- `sudo systemctl status nginx` funciona
- `sudo ls /root` **falla**
- `sudo systemctl restart ssh` **falla**

Guarda toda la evidencia en:

`~/practica3/entrega/32_sudo_webops.txt`

<Aside type="caution" title="Regla LFCS">
No edites `/etc/sudoers` a mano. Usa `visudo` (y valida siempre antes de salir).
</Aside>

## Ejercicio 3.3 — Gestión de paquetes (apt/dpkg) con verificación y trazabilidad

### Contexto

En LFCS te pedirán instalar/quitar herramientas y demostrar que sabes inspeccionar paquetes, binarios y ficheros que instalan.

### Tareas

1) Instala `curl`, `git` y `htop`.

2) Demuestra con evidencia:

- versión de cada herramienta
- qué paquete provee el binario `curl` (pista: `dpkg -S`)
- lista de ficheros instalados por `openssh-server` o `nginx` (elige uno) usando `dpkg -L`

3) Elimina `htop` y demuestra que ya no está disponible.

Guarda comandos y salidas en:

`~/practica3/entrega/33_paquetes.txt`

## Ejercicio 3.4 — Diagnóstico de red (ip, ss, rutas, DNS) con fallos sembrados

### Contexto

Tu servidor “parece caído”: la web no responde, y la resolución de nombres está manipulada. Debes diagnosticar con **`ip a`**, **`ip r`**, **`ss`**, `systemctl`, y pruebas tipo `ping/curl`.

### Setup — el profesor ejecuta esto antes de darte el servidor

```bash
# (Para el profesor — no ejecutes esto tú)
sudo apt install -y nginx ufw
sudo systemctl stop nginx
sudo ufw --force enable
sudo ufw default deny incoming
sudo ufw allow ssh
echo "192.0.2.1 google.com" | sudo tee -a /etc/hosts
```

### Tareas

1) Sin “adivinar”, recopila un diagnóstico inicial con:

- `ip a`
- `ip r`
- `sudo ss -tulpn`
- `systemctl is-active nginx`
- `sudo ufw status verbose`

Guarda el output en:

`~/practica3/entrega/34_diagnostico_inicial.txt`

2) Repara el sistema para que se cumpla:

- `curl -s -o /dev/null -w "HTTP: %{http_code}\n" http://localhost` devuelva 200
- `ping -c 1 8.8.8.8` funcione
- `ping -c 1 google.com` funcione y no resuelva a `192.0.2.x`
- el firewall permita HTTP (80/tcp)

3) Entrega evidencia final de que está arreglado en:

`~/practica3/entrega/34_verificacion_final.txt`

## Ejercicio 3.5 — SSH con llaves + hardening (sshd_config) y pruebas sin bloquearte

### Contexto

La meta es operar como un sysadmin real: usar llaves, asegurar permisos de `~/.ssh`, y endurecer `sshd_config` sin bloquearte.

### Tareas

1) Genera un par de claves **ED25519** para tu usuario actual (si ya tienes, crea uno nuevo con otro nombre). Guarda:

- el listado `ls -la ~/.ssh/`
- la huella de la clave pública (`ssh-keygen -lf ...pub`)

En:

`~/practica3/entrega/35_claves_locales.txt`

2) Crea el usuario `remoteops` (HOME + bash) y configura acceso por llave:

- Crea `/home/remoteops/.ssh/authorized_keys`
- Copia dentro tu clave pública
- Ajusta permisos correctos (`~/.ssh` 700, `authorized_keys` 600) y propietario

Guarda evidencia en:

`~/practica3/entrega/35_authorized_keys_perms.txt`

3) Endurece SSH en `/etc/ssh/sshd_config` para:

- `PermitRootLogin no`
- `PasswordAuthentication no`
- `PubkeyAuthentication yes`

Aplica cambios con `reload` o reinicio y **prueba** que:

- puedes abrir una nueva sesión SSH contra `localhost` como `remoteops` usando llave
- una autenticación por contraseña queda rechazada

Guarda comandos/salidas en:

`~/practica3/entrega/35_hardening_pruebas.txt`

<Aside type="caution" title="No te bloquees">
Antes de desactivar `PasswordAuthentication`, asegúrate de que el login por llave funciona (abre otra terminal y prueba). Si te equivocas, podrías perder el acceso remoto.
</Aside>
