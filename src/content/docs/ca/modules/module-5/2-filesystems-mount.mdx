En la lliçó anterior vam tallar el disc nou i vam aconseguir tenir un bloc físic anomenat `/dev/sdb1`. Però aquest bloc està completament cru; és un tros d'acer sense indexar. Si intentes desar un fitxer de text en ell, el programari no sap en quin racó ficar-lo.

Per organitzar els bytes en carpetes llegibles, cal **formatar-lo** estampant-li un "Sistema de Fitxers" (Filesystem).

---

## 1. El Pinzell de Format: Família `mkfs`

En l'ecosistema oficial Linux empresarial existeixen dos Sistemes de Fitxers reis i indiscutibles. Escullis el que escullis, el comandament creador sempre començarà amb Make Filesystem (`mkfs`).

- **ext4**: L'estàndard tot terreny de Debian. Segur, estable i amb un sistema de 'journaling' (diari) que evita que es corrompi si se'n va la llum de cop.
  ```bash
  sudo mkfs.ext4 /dev/sdb1
  ```
- **xfs**: El rei absolut de RedHat (i molt suportat a Debian). Està dissenyat purament per gestionar eficientment milions de fitxers gegants en paral·lel usant centenars de fils de CPU.
  ```bash
  sudo mkfs.xfs /dev/sdb1
  ```

_(Aquests comandaments destruiran immediatament totes les dades antigues que existissin en aquesta partició. No hi ha advertència. Fes-ho bé)._

---

## 2. L'Empelt Màgic: El comandament `mount`

A Windows, en formatar, et surt una lletra màgica `E:`. Això no existeix a UNIX.
A Linux només existeix una única arrel omnipotent `/`. Si vols usar el teu nou disc `/dev/sdb1`, has de construir una carpeta buida genèrica a l'arrel, i "enganxar" el disc per darrere tapant aquesta carpeta. Això s'anomena **Muntar**.

Imagina un servidor web. El disc principal té el sistema operatiu, però el teu cap t'ha comprat el disc `sdb1` de 5TB per ficar-hi les fotos de Nginx.

```bash
# 1. Crear la carpeta buida receptacle (Punt de Muntatge)
sudo mkdir -p /var/www/fotos

# 2. Enganxar/Muntar el disc a sobre del tapet buit temporalment a la memòria
sudo mount /dev/sdb1 /var/www/fotos
```

En aquest moment, qualsevol cosa que llencis a `cp foto.jpg /var/www/fotos/` s'està desant físicament dins del metall de `sdb1`.

_(Per desconnectar-lo en calent com si fos un USB extraïble, usa `sudo umount /var/www/fotos`)_.

---

## 3. La Immortalitat i el Perill: `/etc/fstab`

El comandament `mount` anterior és amnèsic. Funciona en RAM. Si el servidor s'apaga avui per la nit i reinicia, amanirà amb la carpeta `/var/www/fotos/` completament buida i els clients web escopiran errors 404.
El disc segueix allà, amb les dades vives a dins, però Linux no s'ha recordat de muntar-lo automàticament en la Fase 5 de l'arrencada.

Per immortalitzar la decisió l'hem d'anotar a la sagrada **File System Table (`/etc/fstab`)**.

### El Problema dels Noms `/dev/sdb1` (L'ADN)

MAI usis `/dev/sdb1` a l'fstab. Si demà endolles un USB abans d'arrencar, el kernel can canviar-se els noms anomenant el servidor `sdc1` i el teu USB `sdb1`, causant que el teu servidor intenti muntar el pendrive com a servidor web, fallant miserablement l'arrencada i morint.

**Solució**: Tot disc formatat rep un ADN únic inesborrable anomenat **UUID** (Unique Universal IDentifier). Imprimeix-lo:

```bash
# Block ID (Treu l'ADN llarg del disc)
sudo blkid
```

Copia el llarguíssim UUID del teu `sdb1` (ex. `bc92...11f`) al teu porta-retalls.

### Editant `/etc/fstab`

```bash
sudo nano /etc/fstab
```

Al final d'aquest fitxer delicat, afegeix una línia amb aquest format estricte (separat per espais o TABs):

| 1. Quin disc (UUID) | 2. On Muntar (Punt) | 3. Format | 4. Opcions Especials | 5. Dump | 6. Chequeig pass |
| ------------------- | ----------------------- | ---------- | ---------------------- | ------- | --------------- |
| `UUID=bc92...11f`   | `/var/www/fotos`        | `ext4`     | `defaults`             | `0`     | `2`             |

- **Opcions especials (`defaults`)**: Li dóna permisos de lectura/escriptura normals.
- **Pass (`2`)**: Li diu al kernel "No sóc el disc arrel (que seria un 1), però chequeja'm matemàticament a la recerca d'errors la propera vegada que reiniciïs abans d'engegar-me".

<Aside type="caution" title="La prova de foc (Vital per a LFCS)">
Si t'equivoques en la sintaxi d' `fstab` (ex. has posat un espai de menys, o t'ha faltat una lletra del UUID), la propera vegada que reiniciïs **el Servidor moriria amb un Kernel Panic atroç irremeiable**.

La prova que salva carreres: Després de desar el Nano i ignorant si reiniciarà bé o no, executa **`sudo mount -a`**. Aquest comandament força Linux a llegir el fitxer i muntar el que no està muntat. Si s'executa en silenci sense donar gens de text ¡Enhorabona! Has editat fstab sense errors de sintaxi i el disc s'enganxarà eternament. Si et dóna error de línia, corre a obrir `fstab` de nou.

</Aside>
