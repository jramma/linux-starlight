## 1. El Triomf de Nginx

Històricament, el rei d'internet era "Apache". Apache funcionava contractant un "cambrer" (Procés) per cada client que entrava a la teva pàgina web. Si entraven 10.000 clients alhora (Efecte Slashdot), Apache contractava 10.000 cambrers, rebentant literalment la memòria RAM del teu servidor i forçant un reinici letal.

**Nginx** és un servidor asintrònic impulsat per "Esdeveniments". Utilitza un sol cambrer matemàtic (Worker) capaç de servir àgilment les comandes de 10.000 clients simultanis sense suar ni parpellejar i consumint 3 mísers Megabytes de RAM. Per això Nginx és l'estàndard daurat al núvol modern LFCS.

### Desplegament del Dimoni

```bash
# Sincronitzem orígens i instal·lem
sudo apt update
sudo apt install nginx

# Ens assegurem que arrenqui si el PC pateix una apagada
sudo systemctl enable --now nginx

# Comprovem el seu estat viu i llegim els seus últims 10 logs
sudo systemctl status nginx
```

## 2. Perforant la Muralla (Integrant Mòdul 7)

Nginx ja està encès i intentant servir la pàgina web de benvinguda a l'univers exterior al port 80... **però el nostre Tallafocs UFW de la classe anterior ho està denegant a nivell Kernel**.

Hem de crear el forat quirúrgic permanent a la muralla per habilitar el tràfic d'internet:

```bash
# Permetem peticions al port estandarditzat Web (HTTP)
sudo ufw allow 80/tcp

# Reiniciem l'armadura
sudo ufw reload
```

Si ara vas al teu navegador i tecleges la IP del teu servidor (`http://tu-ip-publica`), veuràs un gegantí cartell blanc dictaminant "Welcome to nginx!".

## 3. Allotjament Múltiple (Virtual Hosts)

El cartell de benvinguda és bonic, però les empreses no paguen servidors per tenir 1 sola pàgina web. Un sol servidor Debian amb Nginx pot allotjar _infinites pàgines web diferents_ posseint una IP única, discernint intel·ligentment la web a despatxar depenent del nom de domini (URL) que hagi escrit el client a la seva barra de Google Chrome.

A aquesta màgia divisòria se la coneix com a "Blocs Server" o Host Virtual.

Anem a crear la casa de la nostra primera web fictícia anomenada `mi-empresa.com`.

### 1. Forjant la carpeta de documents

```bash
# Creem la carpeta esquelet per als HTMLs
sudo mkdir -p /var/www/mi-empresa.com/html

# Li regalem la propietat de la carpeta a l'usuari fosc mestre web (Mòdul 2 chown)
sudo chown -R www-data:www-data /var/www/mi-empresa.com/html

# Escrivim un diminut HTML de prova
echo "<h1>La web de l'empresa</h1>" | sudo tee /var/www/mi-empresa.com/html/index.html
```

### 2. Ensenyar-li els plànols a Nginx

Nginx té una carpeta on desa totes les webs "disponibles". Hem de crear-li un perfil:

```bash
sudo nano /etc/nginx/sites-available/mi-empresa.com
```

Dins, escriurem el seu perfil estricte indicant-li a quin domini atén i en quina carpeta del disc dur resideix.

```nginx
server {
    listen 80;
    server_name mi-empresa.com www.mi-empresa.com;

    root /var/www/mi-empresa.com/html;
    index index.html;
}
```

### 3. L'Enllaç Simbòlic i el Reinici (Integrant Mòdul 2)

Nginx separa elegantment els llocs "Disponibles" (`sites-available`) dels que de debò estaran "Encesos al públic" (`sites-enabled`).
Per encendre la nostra web, Sysadmin LFCS mai copia el fitxer. Crea un Accés Directe Inqüestionable (Enllaç Simbòlic suau `ln -s`) apuntant des de la carpeta de disponibilitat cap a la carpeta d'encesa.

```bash
# Forjant l'enllaç simbòlic
sudo ln -s /etc/nginx/sites-available/mi-empresa.com /etc/nginx/sites-enabled/

# El comandament VITAL: Nginx revisa les teves faltes d'ortografia a nano
sudo nginx -t

# Si el test diu "Syntax OK", recarreguem Nginx pacíficament en calent
sudo systemctl reload nginx
```

Felicitats. Acabes d'allotjar una web professional nivell LFCS.

2. Hem ficat una plantilla bellíssima corporativa de Bootstrap plena de css i imatges al flamant Document Root `/var/www/superempresa/html`. Però al entrar des del navegador, en comptes de veure la web, obtenim un gegantí codi HTTP `"403 Forbidden"` (Prohibit el pas). Sabent que tu vas iniciar sessió com a root, vas crear la carpeta com a root i vas ficar els fitxers per SSH com a root... Per què el motor asintrònic Nginx s'oposa obstinadament a llegir i despatxar a l'exterior el contingut d'aquella carpeta a l'internauta?

   <MultipleChoice>
     <Option>
       Perquè Nginx requereix que converteixis tots els HTMLs a executables amb
       `chmod +x`.
     </Option>
     <Option isCorrect>
       Col·lisió de la Política Discrecional (DAC). Tu vas crear la carpeta i els
       htmls, per tant l'amo dels fitxers és `root` o el teu usuari
       personal. PERÒ el dimoni treballador `nginx` a Debian NO s'executa com a
       root (està prohibit), s'executa operativament assumint la inofensiva
       identitat del fantasma `www-data`. Nginx (`www-data`) va anar a llegir la
       carpeta de root mancant de privilegis per llegir els fitxers d'un
       usuari superior i va ser rebutjat. Has de regalar-li la tutela estricta
       a Nginx tirant `chown -R www-data:www-data /var/www/superempresa/html`.
     </Option>
     <Option>
       El Tallafocs UFW et va tallar la lectura per culpa de no tenir la regla ssh
       activada.
     </Option>
   </MultipleChoice>
