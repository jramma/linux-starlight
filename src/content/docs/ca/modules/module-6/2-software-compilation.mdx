Et contracten en una empresa que utilitza un programari matemàtic propietari dissenyat el 1999. És l'espina dorsal del negoci, però el programari és tan antic o fosc que **mai ha existit en un repositori de Debian**.
Si executes `apt search mi-programari-rar`, et retornarà zero resultats. Estàs sol.

El creador només et proporciona un fitxer comprimit que conté text pur: el codi font (generalment escrit en llenguatge `C`).  
Perquè el processador l'entengui i Linux guanyi un `comandament` nou, seràs tu el responsable absolut de **construir i acoblar els engranatges** transformant textos llegibles en binaris mecànics. Això és compilar.

---

## 1. Desempaquetar el Tarball

El codi font sol venir empaquetat i ultra-comprimit en el que els informàtics més madurs denominen "Tarball" (pilota de quitrà, usualment `.tar.gz`).
Hem de descomprimir la carpeta font al nostre disc dur de treball (sol ser `/usr/local/src` o `/opt`).

```bash
# Entrar a la zona d'extraccions locals
cd /usr/local/src

# Desempaquetar extreient (eXtract) i imprimint la Verbosa (V) vista de Fitxers (F)
sudo tar -xvf programari_fosc-1.0.tar.gz

# Entrem a la carpeta que acaba de néixer
cd programari_fosc-1.0/
```

A dins, normalment veuràs desenes de fitxers crus com `main.c`, `header.h`, i un parell de fitxers sagrats de text que SEMPRE has de llegir primer: el `README` i l' `INSTALL`.

---

## 2. La Santíssima Trinitat (GNU Build System)

Excepte que el README indiqui instruccions rares, el 90% del codi en `C/C++` de la història d'Unix es compila executant seqüencialment aquests tres passos inexcusables dins la carpeta font:

### L'Arquitecte (`./configure`)

El codi font no està llist per al teu PC exacte.
Al executar aquest petit script que ve a la carpeta, l'Arquitecte escanejarà el teu sistema cercant quin compilador tens, quanta RAM i quin processador Intel o AMD uses.

```bash
./configure
```

- ¿Què sol fallar aquí?: Sol morir al minut 1 escopint `error: gcc compiler not found`. Hauries d'anar corrent a `apt` a instal·lar el paquet `build-essential`. O pot cridar `error: missing libssl`. Cosa que t'obligarà a cercar la llibreria OpenSSL i tornar a llançar el configure.
- Al acabar, construirà els plànols hiperprecisos en un fitxer anomenat `Makefile`.

### Els Obrers Cecs (`make`)

Ara que tenim els plànols, li enviem a les eines constructores genuïnes que comencin a forjar l'acer usant el codi. Construiran l'executable binari, cosa que depenent de la mida del programari podria trigar **segons o catorze hores**.

```bash
# Aixecar l'executable (Si vols que usi els 4 nuclis de la teva cpu usa make -j4)
make
```

### L'Instal·lador Oficial (`make install`)

El pas 2 ha forjat amb èxit el programa, però el programa està secuestrat dins la seva carpeta actual a `/usr/local/src/`.
Hem de donar l'ordre de despiojar els seus executables finalitzats i enviar-los estandarditzadament a la carpeta genuïna del sistema on Linux cerca tots els comandaments (usualment de `local` a `/usr/local/bin`). Com que això toca el cor del sistema, requereix credencials divines.

```bash
sudo make install
```

¡Felicitats! Ara podràs teclejar `programari_fosc` en qualsevol part del teu terminal i funcionarà, escapant per sempre de la ignorància d' `apt`.

## 3. Autòpsia Dinàmica de Llibreries (`ldd`)

El programari compilat ha funcionat mesos... però de cop el servidor s'actualitza amb `apt upgrade` ahir, i avui programes foscos casolans llancen aquest horror:
`error while loading shared libraries: libcrypto.so.1.1: cannot open shared object file`

A Linux, els binaris no sempre vénen amb tot inclòs. Agafen "prestat" d'altres llibreries del sistema en calent. (El que a Windows s'anomenaria `.dll`, a Linux s'anomena fitxers `.so` de _Shared Object_).

Si un programa compilat o fosc no arrenca sense explicar clarament què li falta, utilitza el teu bisturí **List Dynamic Dependencies (`ldd`)** sobre el binari malalt per examinar els seus enllaços trencats:

```bash
ldd /usr/local/bin/programari_fosc
```

Això llançarà un arbre de totes les llibreries a les quals es connecta. Aquelles que el teu programari ja no troba perquè algú, o alguna cosa, les va esborrar o les va actualitzar agressivament per darrere en una altra versió, sortiran brillantment llistades amb el segell letal: `=> not found`. La teva missió LFCS consistirà a viatjar al Repositori o internet i instal·lar específicament el `.so` faltant que demana aquesta autòpsia.

## Comprova els teus coneixements

1. Estàs davant del procés d'instal·lació manual més dur GNU i estàs fart d'estar endevinant. Has tirat `./configure` però el programa acaba el seu cicle emetent un terrorífic missatge: `Configure Error: C compiler cannot create executables`. Sense un compilador universal, no pots avançar al pas `make` perquè no hi ha ningú forjant l'acer de 'C'. ¿Quin mega-paquet legendari del repositori de Debian s'exigeix a tot Sysadmin tenir preparat instal·lat per defecte precisament per solventar aquests dolors en entorns crus sense compilador habilitat?

Però obtens instantàniament el rebuig colossal de bash: `make: *** No targets specified and no makefile found. Stop.`. ¿A què es deu això estructuralment en el format GNU?

   <MultipleChoice>
     <Option>Perquè l'has de llançar sempre sent root (`sudo make`).</Option>
     <Option>

3. Un desenvolupador del teu backend t'envia un programari màgic autocompilat `.exe` d'UNIX. El poses a `/usr/bin/miscript` i intentes executar-lo tirant al shell `./miscript`. Obtens un lacònic error invisible de "Fitxers o Objectes Compartits no Trobats". Necessiteu fer-li una radiografia a aquesta caixa negra empaquetada binària avui mateix per deduir cecament a quines llibreries s'està volent enllaçar dins del teu servidor i quines estan faltants. ¿Quin d'aquests és un pas estàndard per a aquest fi?
