## Exercici 3.1 — Gestió d'usuaris i sudo segur

### Context

En un servidor real mai es treballa com a `root` directament. Es crea un usuari específic per a cada tasca i se li donen només els permisos que necessita. En aquesta pràctica configuraràs l'usuari `deploy` que només podrà reiniciar nginx — res més.

### Setup — executar una sola vegada com a alumne

```bash
# Instal·la nginx si no està instal·lat
sudo apt update && sudo apt install -y nginx

# Verifica que nginx està corrent
sudo systemctl status nginx
```

### Tasca 1 — Crear l'usuari deploy

```bash
# Crea l'usuari amb el seu directori home
sudo useradd -m -s /bin/bash deploy

# Estableix una contrasenya segura (inventa-te'n una que recordis)
sudo passwd deploy
# Escriu la contrasenya dues vegades quan te la demani
```

<Aside type="note" title="Per què -m i -s /bin/bash?">
`-m` crea el directori `/home/deploy` automàticament.  
`-s /bin/bash` assigna bash com a shell (sense això l'usuari no tindria shell interactiva).
</Aside>

### Tasca 2 — Configurar sudo restringit

Volem que `deploy` pugui reiniciar nginx amb sudo però **sense introduir contrasenya** i **sense poder fer cap altra cosa** amb sudo.

```bash
# Crea el fitxer de configuració de sudoers per a deploy
# IMPORTANT: usa visudo o l'editor amb aquest comandament per evitar errors de sintaxi
sudo nano /etc/sudoers.d/deploy
```

Escriu exactament aquesta línia al fitxer:

```
deploy ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart nginx
```

Desa amb `Ctrl+O`, `Enter`, `Ctrl+X`.

Verifica que la sintaxi és correcta:

```bash
sudo visudo -c -f /etc/sudoers.d/deploy
# Ha de dir: /etc/sudoers.d/deploy: parsed OK
```

Prova que funciona (canvia a l'usuari deploy primer):

```bash
# Canvia a l'usuari deploy
sudo su - deploy

# Intenta reiniciar nginx (ha de funcionar sense demanar contrasenya)
sudo systemctl restart nginx

# Intenta llistar fitxers com a root (HA DE FALLAR)
sudo ls /root
# Ha de dir: Sorry, user deploy is not allowed to execute...

# Torna al teu usuari
exit
```

### Tasca 3 — Generar parell de claus SSH per a deploy

L'autenticació per clau SSH és molt més segura que per contrasenya. Anem a configurar-la:

```bash
# Canvia a l'usuari deploy
sudo su - deploy

# Genera un parell de claus Ed25519 (algoritme modern i segur)
ssh-keygen -t ed25519 -C "deploy@servidor-linux"
# Quan demani la ruta, prem Enter per usar la predeterminada (~/.ssh/id_ed25519)
# Quan demani passphrase, prem Enter dues vegades (sense passphrase, per simplificar)

# Afegeix la clau pública al fitxer authorized_keys
mkdir -p ~/.ssh
chmod 700 ~/.ssh
cp ~/.ssh/id_ed25519.pub ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys

# Verifica que existeixen els fitxers
ls -la ~/.ssh/

exit  # Torna al teu usuari
```

### Tasca 4 — Protegir SSH

Edita la configuració d'SSH:

```bash
sudo nano /etc/ssh/sshd_config
```

Assegura't que aquestes línies existeixen i tenen aquests valors (busca amb `Ctrl+W` a nano si no les trobes; si comencen per `#` descomenta-les):

```
PasswordAuthentication no
PubkeyAuthentication yes
PermitRootLogin no
```

<Aside type="caution" title="⚠ Compte abans d'aplicar">
Abans de desactivar l'autenticació per contrasenya, assegura't que el teu usuari `alumne` es pot connectar per clau SSH o seguiràs podent connectar-te per la finestra de VirtualBox directament.
</Aside>

Aplica els canvis:

```bash
sudo systemctl reload sshd
```

### Tasca 5 — Verificar tot junt

```bash
# Verifica els permisos de sudo de deploy
sudo -l -U deploy

# Verifica que root no pot connectar-se per SSH
grep "PermitRootLogin" /etc/ssh/sshd_config

# Verifica el directori .ssh de deploy
ls -la /home/deploy/.ssh/
```

## Exercici 3.2 — Diagnòstic de xarxa: troba les fallades

### Context

Ets el sysadmin. El professor ha "trencat" el servidor de tres maneres diferents. No saps quines ni quantes són. La teva feina és diagnosticar què està fallant i solucionar-ho tot.

### Setup — el professor executa això abans de donar-te el servidor

```bash
# (Per al professor — no executis això tu)
sudo apt install -y nginx ufw
sudo systemctl stop nginx
sudo ufw --force enable
sudo ufw default deny incoming
sudo ufw allow ssh
echo "192.0.2.1 google.com" | sudo tee -a /etc/hosts
```

### Símptomes que experimentaràs

- El servidor web no respon quan fas `curl http://localhost`
- Internet pot tenir comportament estrany

### Metodologia de diagnòstic

Segueix sempre aquest ordre:

**Pas 1 — Comprova si nginx respon:**
```bash
curl -s -o /dev/null -w "HTTP: %{http_code}\n" http://localhost
```
Si no respon (error de connexió), el servidor està parat o el tallafocs el bloqueja.

**Pas 2 — Comprova si nginx està corrent:**
```bash
systemctl is-active nginx
```
Si diu `inactive`, arrenca'l:
```bash
sudo systemctl start nginx
```
Torna al Pas 1. Si ara funciona, hi ha més problemes?

**Pas 3 — Comprova el tallafocs:**
```bash
sudo ufw status verbose
```
Està actiu? Permet el port 80 (HTTP)?  
Si no permet el 80:
```bash
sudo ufw allow 80/tcp
```

**Pas 4 — Comprova la resolució DNS:**
```bash
ping -c 2 google.com
```
Falla o va a una IP inesperada? Inspecciona el fitxer hosts:
```bash
cat /etc/hosts
```
Hi ha alguna línia sospitosa? Una IP com `192.0.2.X` és una IP de documentació que no existeix a internet.  
Elimina la línia falsa:
```bash
sudo nano /etc/hosts
# Esborra la línia amb la IP falsa de google.com
# Desa amb Ctrl+O, Enter, Ctrl+X
```

**Pas 5 — Verifica que tot està resolt:**
```bash
curl -s -o /dev/null -w "HTTP: %{http_code}\n" http://localhost
ping -c 2 google.com | grep "transmitted"
sudo ufw status | grep "80/tcp"
```
